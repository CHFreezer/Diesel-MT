# Tokenizer 训练数据准备指南

## 1. 目的与适用范围

本文档规定 Diesel-MT 最终 tokenizer 训练数据的准备原则、数据组成、清洗、抽样、留出和验收方法。

适用目标：

- 最终基线词表：64k；
- 目标模型规模：约 200M 参数的多语机器翻译模型；
- 目标语言：
  - `eng_Latn`：英文；
  - `zho_Hans`：简体中文；
  - `zho_Hant`：繁体中文；
  - `jpn_Jpan`：日文；
  - `kor_Hang`：韩文。

32k 和 48k tokenizer 用于 MVP 阶段验证训练、保存、重载、评测和部署流程，不代表最终词表大小决策。当前 50% 语料的 32k 候选已经达到四语模型训练的基础门槛，详见第 8 节；项目决定在真正冻结 MVP tokenizer 前补入原生繁体、收紧一次通用清洗规则并执行一次有界重训。计划中的 64k 最终基线仍需在五语数据和独立 holdout 上重新验证，不能仅凭词表更大直接判定更优。本文档不要求立即执行训练，也不修改当前正在运行的训练任务。

## 2. 核心原则

最终 tokenizer 语料应满足以下原则：

1. **代表性优先于总量**：扩大单一 shard 的前缀不能代替跨来源、跨域名、跨时间和跨文体的覆盖。
2. **质量优先于全量**：不以“喂完所有可用文本”为目标，通过规模递增实验判断质量是否已经饱和。
3. **五语显式建模**：简体、繁体必须作为两个独立语言分布准备，不使用简转繁数据代替原生繁体语料。
4. **训练与评测隔离**：tokenizer holdout 必须在语料生成阶段按文档、域名或 shard 隔离，永不进入 trainer。
5. **与推理行为一致**：训练数据采用与线上编码一致的 Unicode、空白和规范化策略。
6. **可复现、可审计**：固定数据版本、来源、抽样算法、随机种子和内容哈希。

## 3. 当前项目语料的主要缺陷

### 3.1 缺少繁体中文

当前语料管线只生成英文、简体中文、日文和韩文，虽然 tokenizer 特殊 token 中已经保留 `zho_Hant`，但没有繁体中文训练文本，也没有繁体中文独立评测。

这会导致繁体文本虽然可能依靠单汉字被编码，却缺少原生繁体词频、固定搭配和 subword merge，序列可能明显长于简体中文。

日文语料包含大量汉字，其中一部分与繁体字形相同或相近，可以帮助基础字符覆盖，但不能替代原生繁体语料。日语使用自己的词汇、假名混写、新字体/旧字体和搭配分布，无法提供普通话繁体所需的词频与 merge 信号。

### 3.2 每种语言只使用一个 shard 的固定前缀

当前配置的 `max_shards_per_language: 1` 会让语料集中在一个高 WDS shard 的局部前缀。该方式便于下载和复现，但不能充分覆盖域名、网页类型、crawl 时间和语言风格。

### 3.3 文档级质量不能保证段落级质量

当前随机样本已观察到：

- 博彩、dating 和 SEO 关键词注入；
- mojibake 和错误编码文本；
- 无关句子或网页片段被拼接到一个长段落；
- 过长网页正文、URL、数字串或重复模板。

HPLT 的 WDS 是重要的文档级质量信号，但在将文档拆成段落后，必须再次执行段落级清洗。

### 3.4 近似去重覆盖不完整

当前项目侧 MinHash 仅覆盖英文和简体中文。HPLT 的全局去重以文档为主，文档拆分后仍可能出现重复页眉、转载段落、模板和近似重复内容。因此五种语言都需要统一的段落级 exact 和 near-duplicate 去重。

### 3.5 语料来源与翻译任务分布不完全一致

当前数据全部来自通用网页。它可以作为 tokenizer 主语料，但不一定覆盖后续 MT 模型实际训练和部署中的新闻、百科、技术文本、对话、产品说明、专名和混合语言输入。

### 3.6 预先固定每语种 10 亿字符，缺少规模饱和实验

训练内存主要受唯一预切分单元和文本噪声影响，而不是只由句子数决定。直接把每语种 10 亿字符作为最终目标，可能只是在放大网页长尾和垃圾单元。最终规模应由独立 holdout 上的增益决定。

### 3.7 评测文本来自训练语料池

当前固定评测集从同一份 MVP corpus 中抽取。全量训练时，这些文本会成为训练内样本，不能作为最终 tokenizer 选型的独立依据。

### 3.8 人工抽检不足

当前每语种 20 条稳定样本主要用于自动文字系统比例检查，数量和审查维度不足以发现域名偏置、SEO 注入、乱码和长段落拼接。

## 4. 推荐的数据源组成

### 4.1 主来源：HPLT 3.0

HPLT 3.0 继续作为主来源。它提供文档质量排序、语言识别、来源 URL、crawl 信息和其他元数据，并且已经区分：

| 项目语言 | HPLT 3.0 来源 |
| --- | --- |
| `eng_Latn` | `eng_Latn` |
| `zho_Hans` | `cmn_Hans` |
| `zho_Hant` | `cmn_Hant` |
| `jpn_Jpan` | `jpn_Jpan` |
| `kor_Hang` | `kor_Hang` |

HPLT 3.0 的 `cmn_Hant` 已包含充足的原生繁体数据，优先直接接入，不需要用简体转换生成主语料。

HPLT 4.0 当前发布的是更接近原始处理池的数据，官方说明其本身不直接适合语言工程。最终清洗版未锁定前，项目继续使用可审计的 HPLT 3.0，不应仅因为版本号更高而切换。

### 4.2 补充来源

建议的初始来源比例：

| 来源 | 建议比例 | 作用 |
| --- | ---: | --- |
| HPLT 3.0 高质量多 shard Web 文本 | 60%–75% | 提供广泛语言、专名和长尾覆盖 |
| Wikimedia/Wikipedia | 10%–20% | 提供相对规范的百科、知识和长文本 |
| 实际 MT 训练语料的源端与目标端 | 15%–25% | 使 tokenizer 与后续训练、部署领域一致 |

平行对齐关系对 tokenizer 训练不是必需的。可以把平行语料两端分别作为对应语言的单语文本，但仍需防止评测集污染。

如果最终 MT 数据尚未锁定，不应过早冻结最终 tokenizer。

## 5. HPLT 抽样方法

### 5.1 多 shard 分层抽样

- 从 WDS 8、9、10 三个质量区间选择多个 shard；
- 大语种至少覆盖多个 shard、多个域名和多个 crawl 时间段；
- `cmn_Hant` 应至少覆盖可用的 WDS 10/9/8 shard；
- 不再只扩大单个压缩文件的头部前缀；
- 使用固定哈希优先级或 reservoir sampling，在流式扫描中确定性抽取文档和段落；
- manifest 记录每个 shard 的 URL、版本、范围、哈希、扫描量和采样量。

### 5.2 域名和来源限额

为避免少数 SEO、新闻转载或模板站点控制词频，应按 eTLD+1 或 HPLT URL 元数据设置单域名上限，并记录最终域名分布。

对于同一来源的大量转载内容，除近似去重外，还应限制单站点、单 crawl 或单文档模板的最大贡献。

## 6. 语言配比

若五种语言在产品中同等重要，首个候选可使用五语各 20% 的训练字符预算。

若后续 MT 训练数据分布已经确定，最终采样率可按下式平滑：

```text
q(language) ∝ p(language) ^ alpha
alpha 建议从 0.3–0.5 试验
```

其中 `p(language)` 是该语言在模型训练数据中的原始占比，`q(language)` 是 tokenizer 训练占比。该方法可防止高资源语言垄断词表，也避免极低资源语言被机械放大。

繁体中文可以从总语料的 15%–20% 开始。它与简体中文、日文共享大量汉字，不必机械地额外准备 10 亿字符；最终比例由繁体 holdout 上的 fertility、序列长度和覆盖率决定。

不得把 `zho_Hans` 和 `zho_Hant` 合并为一个未标注的中文池，也不得把粤语繁体 `yue_Hant` 当作普通话繁体 `zho_Hant`。

## 7. 规模递增方案

最终 64k tokenizer 先构建三个可复现数据档位：

| 档位 | 每语种清洗后字符数 | 五语合计 |
| --- | ---: | ---: |
| S | 50M | 250M |
| M | 100M | 500M |
| L | 200M | 1B |

在完全相同的 64k tokenizer 配置下依次训练，比较：

- 每语言 tokens/character 或 fertility；
- P50/P95/P99 序列长度与下游截断率；
- `<unk>`、原文字符丢失和精确 roundtrip；
- 词表利用率和低频 token 比例；
- 相邻规模之间的 vocabulary/merge 稳定度；
- 五语之间以及平行翻译文本之间的 token 长度差异。

如果 M 到 L 的改进已接近测量噪声，则停止扩容。只有 L 仍有明确收益时，才继续测试更大语料。正式规模由质量饱和曲线决定，不由“全量”标签决定。

## 8. 当前 50%/32k 实测与模型训练门槛

### 8.1 2026-07-14 实测结果

当前 50% 语料的 32k 候选使用约 19.98 亿字符、576 万行四语文本训练，最终总词表为 32,768。使用项目固定评测集（每个训练语种 500 条，共 2,000 条）进行只读评测，结果如下：

| 指标 | 50%/32k 结果 |
| --- | ---: |
| tokens/非空字符 | 0.5820 |
| `<unk>` token 比例 | 0.001948% |
| 原文字符丢失率 | 0.001275% |
| 字符频率加权覆盖率 | 99.998725% |
| 唯一字符覆盖率 | 99.879032% |
| 精确 roundtrip | 99.65% |
| 中日共享汉字覆盖率 | 100% |
| 韩文音节/Jamo 频率和唯一覆盖率 | 100% |
| 英文平均 pieces/word | 1.7539 |

该候选还通过了内存中的微型 `M2M100ForConditionalGeneration` forward 验证：`len(tokenizer)`、模型 `vocab_size`、encoder/decoder embedding 行数和 `lm_head` 输出维度均为 32,768，forward loss 为有限值。

作为规模饱和参考，10%/32k 的 tokens/非空字符为 0.5821，原文字符丢失率为 0.001700%。增加到 50% 后，压缩效率几乎不变，字符丢失率小幅下降。这表明在当前四语数据分布和 32k 词表容量下，10% 到 50% 的主要收益来自罕见字符稳健性，序列压缩已经基本进入收益递减区。继续扩大相同分布的语料不应被视为启动模型 MVP 的前置条件。

### 8.2 当前结论和适用边界

当前 50%/32k 候选已经达到**启动英文、简体中文、日文、韩文四语 MVP 模型训练**的工程门槛。32k 可以作为可靠回退候选，证明 tokenizer 训练、保存、重载和模型接口路线成立。

但项目的 MVP 范围已经包含 `zho_Hant`。鉴于 HPLT 3.0 原生 `cmn_Hant` 可直接接入，现有下载、锁定、流式清洗和训练 checkpoint 机制均可复用，项目决定不把当前四语候选作为最终 `mvp-tokenizer-v0` 冻结，而是在学生模型正式训练前完成一次五语数据升级和一次有界重训。这是产品语言覆盖修正，不是因为当前四语 tokenizer 无法训练模型。

该结论不等于最终 tokenizer 已经冻结，仍存在以下边界：

- 当前语料和固定评测集没有原生繁体中文；`zho_Hant` 语言 token 存在不代表已经验证繁体编码效率和翻译能力；
- 固定评测集来自当前 MVP corpus 池，可用于同条件候选比较，但不能替代最终独立 holdout；
- 48k 候选仍需在同一固定评测集上比较序列长度、字符丢失和模型吞吐；
- CTranslate2 转换和 CPU 推理属于部署验收，不是开始模型训练的覆盖率门槛，但应在 MVP 发布前完成；
- tokenizer 一旦用于正式模型训练，必须冻结完整产物、配置、语言映射、训练 manifest 和 SHA-256；后续更换词表需要重新训练模型。

### 8.3 Tokenizer 在约 200M 翻译模型中的重要性

Tokenizer 重要，但它的收益具有明显门槛和递减效应。需要优先保证的硬条件是：

1. 目标文字不会以不可接受的比例进入 `<unk>` 并丢失；
2. 各语言的序列长度和截断率可控；
3. 特殊 token、语言 token、词表 ID 和模型 embedding/output 维度完全一致；
4. 保存、重载、批量编码和目标语言控制稳定。

达到这些条件以后，翻译质量通常更多取决于平行语料与 HY-MT2 蒸馏译文质量、训练 token 数、语言方向采样、模型容量和优化稳定性。Tokenizer 本身不会赋予模型翻译能力，不应为了微小的 tokenizer 指标改进长期阻塞主模型 MVP。

词表大小仍会显著占用小模型的参数和计算预算。对共享 token embedding，矩阵参数量近似为 `vocab_size × d_model`：

| 词表 | `d_model=512` | `d_model=768` |
| ---: | ---: | ---: |
| 32k | 16.78M | 25.17M |
| 48k | 25.17M | 37.75M |
| 64k | 33.55M | 50.33M |

即使输入 embedding 与输出权重绑定，生成时的输出 logits 维度仍随词表大小增加。因此，更大词表需要用更短序列、更低截断率或更好下游质量来证明其额外成本合理。

### 8.4 32k、48k 和 64k 的决策方法

- **32k**：已满足四语 MVP 门槛，是五语重训失败时的工程回退和后续对照基线。
- **48k**：50% 四语实测比 32k 降低 14.89% tokens/非空字符，P95/P99 分别缩短 14.92%/15.94%，而字符丢失率和覆盖率完全相同；选定为五语一次性重训的默认词表规模。
- **64k**：作为最终基线候选保留，必须在加入原生繁体、改进数据来源和建立独立 holdout 后，与 48k 在相同数据、模型和计算预算下比较。

候选选择采用以下暂定规则：

1. 48k 相比 32k 若缩短至少约 10% 序列，且字符丢失率、各语言覆盖率没有实质恶化，则优先进入模型吞吐实测；
2. 若序列缩短低于约 5%，优先选择参数和 softmax 成本更低的 32k；
3. 若收益处于 5%–10%，用相同 batch、最大长度和硬件直接比较训练 tokens/s、峰值显存、截断率和验证集翻译指标；
4. 64k 不因“最终基线”命名自动胜出，必须证明相对 48k 的净收益。

为缩短 HY-MT2 蒸馏 MVP 的关键路径，教师译文生成可以在学生 tokenizer 冻结前并行开始：保存原始 UTF-8 源文本、目标文本、语言方向、教师模型版本、prompt 版本和生成参数，待 tokenizer 选定后统一编码。真正开始学生模型训练前再冻结 tokenizer 即可。

### 8.5 MVP 五语一次性重训决定（2026-07-14）

当前 50%/48k 四语候选已经完成。执行以下一次性升级；该升级完成后即冻结 MVP tokenizer，不继续进行无边界的语料和规则调参：

1. 当前同源对比已经选定 48k：四语总体序列缩短 14.89%，各语种缩短 12.33%–16.20%，字符丢失、频率覆盖、唯一字符覆盖和 roundtrip 与 32k 相同。五语数据准备完成后只重训 48k；32k 保留为工程回退和模型吞吐对照。
2. 接入 HPLT 3.0 `cmn_Hant -> zho_Hant`，使用原生繁体，不使用简转繁或日文语料代替。
3. 将 MVP corpus 改为五语平衡语料。推荐每语种约 100M–200M 个清洗后字符，首选 200M、五语合计约 1B；不继续沿用每语种 500M 的当前 50% 规模。现有 10% 到 50% 的饱和结果已经说明扩大同分布语料的收益很小，而较小的五语语料可显著降低 checkpoint 内存和重训时间。
4. 将清洗算法版本升级一次，按第 10 节收紧段落、乱码、模板、机械长串、异常比例和五语去重规则；保留各过滤原因计数和人工抽检，避免静默过度过滤。
5. 为 `zho_Hant` 建立独立固定评测集和压力样例，至少报告 tokens/字符、P95/P99、字符丢失、唯一字符覆盖、roundtrip，以及与 `zho_Hans` 的序列长度差异。
6. 重新跑保存/重载、五语 language token、微型 M2M100 forward 和最终产物 SHA-256 验证，然后冻结为 `mvp-tokenizer-v0`。

本轮不训练当前四语数据的 64k tokenizer。64k 留到多来源五语最终数据准备完成后评估，以免额外词表容量学习当前网页噪声，也避免继续阻塞 HY-MT2 蒸馏 MVP。

现有实现并非只需修改配置文件。以下四语假设必须同步泛化并补测试：

- 数据管线中的固定 `LANGUAGES`、启用 source/profile 必须恰好四语的校验；
- `script_ratio()` 对简体、繁体的区分逻辑，以及五语质量阈值；
- 当前仅覆盖英文和简体中文的近似去重白名单；
- `TRAINING_LANGUAGES`、语料 manifest、round-robin 平衡、固定评测集、专项指标和测试 fixture。

下载、source lock、断点缓存、流式解压、通用清洗、manifest、Rust feed checkpoint 和模型语言 token 可以继续复用。因此该工作不是零成本配置切换，但边界明确，不需要重写训练器。

## 9. 分段与文本规范

### 9.1 分段

推荐流程：

1. 从 HPLT 文档中读取 paragraph-like segment；
2. 对过长 segment 按换行和句末标点继续切分；
3. 无法自然切分时，按 Unicode 边界分块；
4. 建议单块不超过约 2,048 Unicode 字符或 4,096 UTF-8 字节，先达到哪个上限就切分；
5. 超长无空格拉丁串、URL、Base64、hash 和机械重复串不应作为正常段落进入 trainer。

### 9.2 Unicode 和简繁处理

- 输入必须是有效 UTF-8；
- 训练和推理使用相同的 Unicode 规范化策略；
- 不将原生繁体统一转为简体，也不将简体统一转为繁体；
- 简转繁数据只可作为受控增强，不能替代原生繁体；
- 保留真实标点、数字、Emoji、专名和自然 code-switching，但限制异常集中来源；
- 明确记录全角/半角、兼容字符和空白折叠策略。

## 10. 段落级清洗

每个候选段落至少经过：

- 文档级和 segment 级语言识别；
- 目标文字系统比例检查；简体和繁体除汉字总比例外，还需用可靠的简繁特征字或语言识别信号检查明显错分，但不得删除两边自然共享的汉字；
- HTML、模板、导航、cookie、页眉页脚残留检查；
- mojibake、替换字符和控制字符检查；除 `U+FFFD` 外，增加对常见 UTF-8/Latin-1 误解码片段的评分检测，并保留命中样例供人工审查；
- SEO、博彩、dating、关键词堆砌和无关插入检测；
- 重复字符、重复 n-gram、异常标点、数字或符号比例检查；
- URL、邮件、Base64、hash、代码和超长无空格单元检测；允许自然技术文本，但拒绝由这些单元主导的段落；
- 长度和可见字符比例检查；过长文本应优先按自然边界分块，而不是保留接近 100k 字符的网页行；
- 页眉、菜单、版权、cookie、登录/注册和站内推荐等跨文档高频模板检查；
- 每域名、单模板和单 crawl 的贡献上限，以及过滤前后来源分布审计。

质量规则需要允许合理的技术文本、URL、代码、数字、混合语言和非 BMP Unicode。目标是过滤不自然的集中污染，而不是把所有边缘输入删除。

MVP 本轮收紧清洗时必须遵守“规则有测试、过滤有原因、样本可审查”的约束：每个新增拒绝规则至少包含正例和反例测试；先在小规模 smoke 上统计各语种拒绝率和人工抽检，再生成正式五语 corpus。不得只扩大垃圾关键词正则并直接重跑全量，否则容易误删新闻、法律、医疗和技术文本中的正常词汇。

## 11. 去重和数据隔离

五种语言统一执行；本轮 MVP 不再只对英文和简体中文启用近似去重：

1. 规范化后段落 exact hash 去重；
2. 段落级 MinHash/SimHash 近似去重；
3. 跨来源去重；
4. tokenizer train 与 tokenizer holdout 之间跨集合去重；
5. 与下游翻译评测集做污染检查。

去重应在分段以后执行。文档级全局去重不能替代段落级去重。

## 12. 独立 holdout

最终语料生成时预留 2%–5% 作为 tokenizer holdout，并满足：

- 按整个文档、域名、crawl 或 shard 隔离，不做随机行级拆分；
- 覆盖五种语言；
- 包含通用 Web、百科、实际翻译领域、混合语言和噪声压力样本；
- 包含五语平行翻译文本，用于测量 token 长度 parity；
- 不进入 Python iterator、Rust feed checkpoint 或任何 tokenizer trainer；
- manifest 单独记录并冻结哈希。

FLORES 等正式翻译评测集不得加入 tokenizer 训练语料。

## 13. 人工质量审查

最终冻结前，每种语言至少人工检查 200–500 条文本，并按以下维度分层：

- 数据源、WDS 和 shard；
- 域名和 crawl 时间；
- 长度区间；
- 正常样本和过滤阈值附近样本；
- 简体、繁体及其混合文本；
- URL、代码、数字、Emoji、专名和 code-switching；
- 被接受和被拒绝样本。

报告必须显示可审查文本或安全截断片段，不能只显示内容哈希和“自动通过”。

## 14. NLLB 简繁语言标识

NLLB 和 Transformers `NllbTokenizer` 已定义：

- `zho_Hans`：普通话/中文，简体字；
- `zho_Hant`：普通话/中文，繁体字；
- `yue_Hant`：粤语，繁体字。

`src_lang` 用于标识编码器输入语言，`tgt_lang` 或生成阶段的目标语言 token 用于标识解码器输出语言。

语言 token 只提供条件标识，不会自动赋予模型该语言的翻译能力。约 200M 的目标模型仍需在带正确语言标签的平行数据上学习简体和繁体的输入、输出方向。

## 15. 加入繁体对约 200M 模型的影响

### 15.1 Tokenizer 层面

- 新增一个语言 token 只增加一个 embedding 行，参数影响可忽略；
- 64k 词表对五种语言具有足够候选容量；
- 繁体与简体、日文共享大量汉字，新增词表容量成本明显低于增加一种全新文字系统；
- 原生繁体数据会占用一部分独有 subword，但能显著改善繁体序列长度和词汇覆盖。

因此，只从 tokenizer 看，合理加入繁体不应导致可见的整体性能下降。

### 15.2 模型训练层面

可能的性能下降主要来自固定训练预算下的容量和 batch 竞争，而不是语言 token：

- 若总训练 token 和步数不变，从四语均分改为五语均分，原四语的平均曝光比例会下降；
- 200M 模型存在多语容量上限，采样失衡可能造成语言间干扰；
- 简繁高度相关，也可能通过共享词汇、句法和语义获得正迁移。

控制方法：

- 使用平滑或温度采样；
- 不让繁体无限放大，也不把它压缩到只能字符级编码；
- 必要时增加总训练 token，而不是完全从原四语中扣除；
- 使用相同架构、数据和计算预算进行四语/五语小规模对照；
- 分别报告原四语和繁体的 chrF/BLEU、截断率与推理吞吐。

预期上，加入繁体的风险可控，且小于完全遗漏繁体所造成的产品能力缺口，但最终结论必须由下游对照实验确认。

## 16. 冻结前验收清单

- [ ] 五种语言均有原生训练语料和独立 holdout；
- [ ] `cmn_Hant` 正确映射为 `zho_Hant`，未与 `yue_Hant` 混淆；
- [ ] HPLT 采样覆盖多个 shard、域名和 crawl；
- [ ] 分段后执行五语 exact 与 near-duplicate 去重；
- [ ] 没有使用 tokenizer holdout 或正式 MT 评测集训练；
- [ ] 完成 S/M/L 规模饱和比较；
- [ ] 完成五语人工分层审查；
- [ ] 64k 候选在五语 holdout 上无不可接受的字符丢失；
- [ ] 记录来源、版本、许可证、抽样算法、随机种子和 SHA-256；
- [ ] tokenizer 一旦用于模型训练，冻结完整产物和训练语料 manifest。
